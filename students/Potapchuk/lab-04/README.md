## Цель работы

Реализовать прикладной слой (Application Layer) системы с разделением операций на **команды** (изменяют состояние домена) и **запросы** (читают данные без изменений) в соответствии с паттерном **CQRS**.

**Вариант:** Request Service (Система обработки заявок / ПСО)  
Ядро домена: Агрегат `Request`, объекты-значения и события из лабораторной работы №3.

---

## Ход выполнения работы

### 1. Команды (Commands)

В прикладном слое созданы иммутабельные объекты команд для изменения состояния системы.

**Созданные команды:**
- `CreateRequestCommand` — создание новой заявки
- `AssignGroupToRequestCommand` — назначение группы на заявку
- `ActivateRequestCommand` — активация заявки
- `ChangeRequestZoneCommand` — изменение зоны заявки
- `CompleteRequestCommand` — завершение заявки

Все команды реализованы как `record`, являются иммутабельными и содержат атрибуты валидации (`[Required]`, `[MinLength]`, `[Range]` и др.).

**Пример кода команды:**
```csharp
public record CreateRequestCommand
{
    [Required, MinLength(5), MaxLength(200)]
    public string Title { get; init; } = string.Empty;

    [MaxLength(1000)]
    public string? Description { get; init; }

    [Required]
    public string RequesterId { get; init; } = string.Empty;

    [Required]
    public string InitialZoneId { get; init; } = string.Empty;

    [Range(1, 5)]
    public int EmergencyLevel { get; init; };
}
```
### 2.Queries (Запросы) и DTO
Для чтения данных реализованы запросы и DTO-модели.
Запросы:
GetRequestByIdQuery — получение заявки по ID
ListActiveRequestsQuery — список активных заявок
Read DTO:
RequestDto — упрощённая модель для передачи данных наружу
### 3. Query Handlers
GetRequestByIdQueryHandler и ListActiveRequestsQueryHandler — отвечают только за чтение данных и маппинг доменной модели в DTO. Состояние системы не изменяется.
### 4. Application Service (Фасад)
Создан класс RequestService, реализующий интерфейс IRequestService, который делегирует выполнение команд и запросов соответствующим handlers.

## Контрольные вопросы

**1. В чём разница между Command и Query?**  
Команда (Command) выражает намерение изменить состояние системы (Create, Update, Delete) и обычно возвращает только ID созданного ресурса. Запрос (Query) предназначен исключительно для чтения данных, не изменяет состояние и возвращает DTO.

**2. Почему Command Handler возвращает только ID, а не весь объект?**  
Это минимизирует передачу данных и разделяет ответственность. Клиент получает ID, чтобы при необходимости выполнить отдельный запрос (Query) для получения актуального представления объекта. Это предотвращает утечку доменной логики в слой представления.

**3. Где должна выполняться валидация?**  
Валидация формата данных (тип, обязательность) — в Команде/Handler. Валидация бизнес-инвариантов (правила системы) — строго внутри Доменной модели.

**4. Можно ли вызывать Query из Command Handler?**  
Технически возможно, но нарушает принцип чистого CQRS. Для получения данных внутри Handler лучше обращаться напрямую к репозиторию.

**5. Зачем разделять Request DTO и Command?**  
Это позволяет независимо изменять внешний API (JSON формат) и внутреннюю логику приложения. Это защищает прикладной слой от изменений в веб-фреймворках.
