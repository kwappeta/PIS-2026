## Цель работы

Заменить REST API на высокопроизводительный протокол **gRPC** с использованием **Protocol Buffers**. Реализовать unary-вызовы и server-side streaming для получения реального времени обновлений.

---

## Выполненная работа

### 1. Protocol Buffers (.proto файл)

Создан файл `request_service.proto`, в котором описаны:
- Service `RequestService`
- Unary RPC: `CreateRequest`, `GetRequest`
- Server-side streaming: `StreamActiveRequests`

### 2. gRPC Server

Реализован сервер на ASP.NET Core gRPC с сервисом `RequestServiceImpl`, содержащим:
- Метод создания заявки (`CreateRequest`)
- Метод получения заявки (`GetRequest`)
- Streaming-метод `StreamActiveRequests` (отправляет обновления каждую секунду)

### 3. gRPC Client

Создано консольное приложение-клиент, которое:
- Вызывает unary-методы
- Подключается к серверному стримингу и выводит данные в реальном времени

### 4. Генерация кода

Код C# был автоматически сгенерирован из `.proto` файла с помощью пакетов:
- `Grpc.AspNetCore`
- `Grpc.Tools`

## Контрольные вопросы

**1. В чём преимущество gRPC над REST?**  
gRPC быстрее REST, использует бинарный формат Protocol Buffers, поддерживает streaming, строго типизированные контракты, HTTP/2 и bidirectional streaming.

**2. Почему Protocol Buffers быстрее JSON?**  
Protocol Buffers — бинарный формат, он компактнее, быстрее сериализуется/десериализуется и занимает меньше места по сравнению с текстовым JSON.

**3. Зачем нужен streaming в gRPC?**  
Streaming позволяет передавать данные частями в реальном времени (например, обновления заявок, логи, мониторинг) без необходимости делать множество отдельных запросов.

**4. Когда использовать REST, а когда gRPC?**  
- **REST** — для публичных API, когда нужен браузер, простота и отладка.  
- **gRPC** — для внутренней коммуникации между микросервисами, где важны скорость, низкая задержка и сильная типизация.
