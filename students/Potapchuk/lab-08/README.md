## Цель работы

Разбить монолитное приложение Request Service на микросервисы по принципу **Bounded Contexts**, организовать асинхронную коммуникацию через Event Bus (RabbitMQ) и настроить API Gateway.

---

## Выполненная работа

### 1. Определение Bounded Contexts

Система разделена на три микросервиса:

| Микросервис           | Bounded Context               | Ответственность |
|-----------------------|-------------------------------|-----------------|
| **Request Service**   | Request Management            | Создание, назначение, изменение статуса заявок |
| **Group Service**     | Group Management              | Управление поисковыми группами и участниками |
| **Notification Service** | Notification Management    | Отправка уведомлений пользователям |

### 2. Реализация микросервисов

- Каждый сервис имеет собственный REST API
- Каждый сервис запускается в отдельном контейнере
- У каждого сервиса своя база данных (в данной работе — упрощённо)

### 3. Event Bus (RabbitMQ)

Настроена асинхронная коммуникация:
- `RequestCreated`
- `GroupAssignedToRequest`
- `RequestActivated`

После назначения группы на заявку Request Service публикует событие, которое получает Notification Service.

### 4. API Gateway (YARP)

Настроен API Gateway, который маршрутизирует запросы:
- `/api/requests/**` → Request Service
- `/api/groups/**` → Group Service
- `/api/notifications/**` → Notification Service

### 5. Docker Compose

Все сервисы + RabbitMQ + API Gateway запускаются одной командой через `docker-compose.yml`.

## Контрольные вопросы

**1. Что такое bounded context?**  
Bounded Context — это явная граница поддомена, внутри которой используется единый язык и модель данных. Каждый микросервис обычно соответствует одному bounded context.

**2. Почему микросервисы не должны делить БД?**  
Общая база данных создаёт сильную связанность между сервисами, нарушает принцип независимого развёртывания и усложняет масштабирование.

**3. В чём проблема распределённых транзакций?**  
Распределённые транзакции (2PC) очень сложны, снижают доступность системы и плохо масштабируются. Вместо них обычно используют Saga Pattern или Event-Driven подход.

**4. Зачем нужен Circuit Breaker?**  
Circuit Breaker защищает систему от каскадных сбоев: если один сервис не отвечает, Circuit Breaker быстро перестаёт отправлять ему запросы и возвращает fallback.
