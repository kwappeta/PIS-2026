## Вариант №19 — Электронная библиотека

**Питч:**  
Система для отслеживания чтения книг, управления прогрессом и выставления оценок.

**Ядро домена:**
- Агрегат: `UserLibrary`
- Сущность: `Book`
- Value Objects: `ReadingProgress`, `Rating`, `BookId`

**Технологический стек:**
- C# / .NET
- xUnit
- Moq
- Testcontainers (PostgreSQL)

## Ход выполнения работы

### 1. Юнит-тесты (Domain Layer)

**Файл:** `BookTests.cs`  
**Покрытие:** инварианты и бизнес-логика сущности `Book`.

**Проверенные сценарии:**
- Нельзя установить прогресс больше общего числа страниц
- Книга автоматически получает статус `Completed` при завершении
- Генерация события `BookCompletedEvent`
- Нельзя завершить книгу дважды

**Пример теста:**
```csharp
[Fact]
public void Should_Set_Status_To_Completed_When_Finished()
{
    var book = new Book(new BookId(Guid.NewGuid()), "Test", "Author", 100);

    book.UpdateProgress(100);

    Assert.Equal("Completed", book.Status);
}
```
### 2. Юнит-тесты (Application Layer)

Файл: AggregateTests.cs
Инструментарий: Moq

Проверяется логика агрегата UserLibrary:

Добавление книги
Запрет на добавление дубликатов
Добавление оценки только после завершения книги

Результат:
Агрегат корректно управляет доступом к сущностям и соблюдает инварианты.

### 3. Интеграционные тесты (Infrastructure Layer)

Файл: RepositoryTests.cs (или аналогичный)

Технология: Testcontainers (PostgreSQL)

Используется реальная база данных в контейнере.

Проверено:

Сохранение книги (Save)
Загрузка книги (GetById)
Корректность маппинга

Почему не In-Memory:

не проверяет ограничения
отличается поведение SQL
нет реальных типов данных

### 4. E2E-тесты (Сквозной сценарий)

Файл: LibraryFlowTests.cs

Инструменты: WebApplicationFactory, HttpClient

Сценарий:

POST /books — создание книги
PUT /books/{id}/progress — обновление прогресса
GET /books/{id} — получение данных

Результат:
Проверена вся цепочка:

Controller → Service → Domain → Repository → Database
### Ответы на контрольные вопросы
1. В чём разница между юнит-тестом и интеграционным?

Юнит-тест проверяет отдельный класс или метод без внешних зависимостей.
Интеграционный тест проверяет взаимодействие компонентов (например, с БД).

2. Почему юнит-тестов больше?
Быстрее выполняются
Проще поддерживаются
Легче находить ошибки
3. Зачем Testcontainers?

Позволяет тестировать с реальной PostgreSQL, а не упрощённой in-memory БД.

4. Что такое Mutation Testing?

Это проверка качества тестов путём внесения изменений в код.
Если тесты не падают — они недостаточно хорошие.

### 5. Вывод

В ходе работы реализована полноценная стратегия тестирования системы «Электронная библиотека».

Юнит-тесты покрывают бизнес-логику и инварианты
Интеграционные тесты проверяют работу с БД
E2E-тесты подтверждают корректность всей системы

Данный подход позволяет:

снизить количество ошибок
упростить поддержку кода
повысить надежность системы
