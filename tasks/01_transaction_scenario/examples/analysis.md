# Анализ границ ответственности

**Система:** Поисково-спасательный отряд «Юго-Запад» 🔦  
**Сценарий:** Создание заявки на поиск и формирование группы

---

## 1. Транзакционные границы

### 1.1. Где начинается и заканчивается транзакция?

**Начало транзакции:** Координатор нажимает кнопку "Создать заявку"

**Конец транзакции (успешный сценарий):**
- **Основная транзакция БД №1:** Заявка сохранена + Зоны созданы
- **Основная транзакция БД №2:** Группа сформирована + Зона назначена (обновление статусов)
- **Побочные эффекты (асинхронные):** SMS и push-уведомления отправлены участникам

**Важно:** У нас две отдельные транзакции:
1. **Создание заявки + разбиение на зоны** - атомарно
2. **Формирование группы + назначение на зону** - атомарно (но может откатиться независимо от п.1)

Если формирование группы не удалось, заявка остаётся в статусе `NEW`, и группу можно создать позже.

### 1.2. Какие операции должны быть атомарными?

#### Транзакция №1: Создание заявки

**Атомарные операции (в одной БД-транзакции):**
1. Создание записи Request в таблице `requests`
2. Валидация координат района (lat, lon, radius)
3. Вызов сервиса геопланирования для разбиения на зоны
4. Создание 4 записей Zone в таблице `zones` (North, South, East, West)
5. Создание связи Request ↔ Zones

**Обоснование:** Заявка без зон не имеет смысла. Если разбиение на зоны не удалось - заявка недействительна.

**Пример сбоя:**
- Если создание заявки удалось, но геосервис вернул ошибку → ROLLBACK всей транзакции
- Иначе получим "заявку-зомби" без зон для поиска

#### Транзакция №2: Формирование группы

**Атомарные операции (в одной БД-транзакции):**
1. Проверка доступности волонтёров (статус `AVAILABLE`)
2. Создание записи Group в таблице `groups`
3. Создание связей Group ↔ Participants в таблице `group_members`
4. Обновление статусов волонтёров на `BUSY`
5. Назначение группы на зону (обновление `zone.assigned_group_id`)
6. Обновление статуса зоны на `ASSIGNED`
7. Обновление статуса заявки на `IN_PROGRESS`

**Обоснование:** Если волонтёр становится занятым, но группа не создалась → получим "зависшего" волонтёра.

### 1.3. Какие операции могут быть асинхронными?

| Операция | Обоснование |
|----------|-------------|
| Отправка SMS старшему группы | Может быть отложена. Группа уже сформирована, SMS - это "best-effort" действие |
| Отправка push-уведомлений волонтёрам | Аналогично SMS - не критично для создания группы |
| Запись событий в журнал событий (Events Log) | Eventual consistency допустима. События можно записать асинхронно через event sourcing |
| Обновление статистики отряда (количество активных заявок) | Можно пересчитать позже |
| Синхронизация с картографическим сервисом (отображение зон на карте) | Не влияет на бизнес-логику |

**Принцип:** Если операция НЕ нарушает инварианты доменной модели при отказе - она может быть асинхронной.

| **Запись событий в журнал событий (Events Log)** | Асинхронная (event sourcing) | Нет (best-effort) | 3 попытки | Да (дедупликация по event_id) |
| **Обновление кэша статистики отряда (Redis)** | Асинхронная | Нет (eventual consistency) | 3 попытки с задержкой 1s | Да |

## 3. Обработка отказов внешних сервисов

### 3.1. Notification Service недоступен

**Проблема:** Сервис SMS/Push-уведомлений не отвечает (таймаут, 503 Service Unavailable)

**Стратегия:**
1. **Таймаут:** 10 секунд (для синхронного вызова) или N/A (для async через очередь)
2. **Основная транзакция:** НЕ откатывается (заявка и группа остаются в БД)
3. **Retry:**
   - Помещаем уведомления в очередь `notification_queue`
   - Фоновый worker пытается отправить повторно
   - Retry schedule: через 1 минуту, затем 5 минут, 15 минут, 1 час, 6 часов
   - После 5 неудачных попыток → помечаем как "failed" и алертим админа
   - Отправляем fallback-уведомление координатору: "SMS не отправлена. Позвони старшему вручную."
4. **Компенсация:** Нет (группа уже сформирована, это нормально)
5. **Уведомление пользователя:** "Заявка создана! Уведомления будут отправлены позже. Можно позвонить старшему вручную."

**Обоснование:** Уведомления - это приятный бонус, но не критично. Старший группы всё равно увидит назначение в системе, и координатор может позвонить вручную.

### 3.2. База данных недоступна

**Проблема:** PostgreSQL не отвечает (connection refused, timeout)

**Стратегия:**
1. **Таймаут:** 5 секунд для connection, 30 секунд для query
2. **Основная транзакция:** ROLLBACK (заявка и группа НЕ создаются)
3. **Retry:**
   - На уровне приложения: 3 попытки с экспоненциальной задержкой (1s, 2s, 4s)
   - Если все 3 попытки не удались → возвращаем 503 Service Unavailable
4. **Компенсация:** Нет (ничего не было создано)
5. **Offline-режим:**
   - Если координатор в поле без интернета → сохраняем заявку локально (IndexedDB/SQLite)
   - При восстановлении связи → синхронизируем с сервером
6. **Уведомление пользователя:** "Не удалось создать заявку. Попробуй ещё раз или свяжись с техподдержкой."

**Обоснование:** БД - это критичный компонент. Без неё заявка не может существовать. Но для полевых условий нужен offline-режим.

### 3.3. Сервис геопланирования недоступен

**Проблема:** Сервис разбиения территории на зоны не работает (или вернул ошибку)

**Стратегия:**
1. **Таймаут:** 15 секунд
2. **Основная транзакция:** ROLLBACK (заявка НЕ создаётся без зон)
3. **Retry:**
   - 3 попытки с задержкой 500ms
   - Если не удалось → возвращаем 500 Internal Server Error
4. **Компенсация:** Нет (заявка без зон бесполезна)
5. **Fallback:**
   - Предлагаем координатору создать зоны вручную через интерфейс карты
   - Сохраняем заявку в статусе "DRAFT" до ручного определения зон
6. **Уведомление пользователя:** "Не удалось автоматически разбить район на зоны. Создай зоны вручную на карте."

**Обоснование:** Зоны критичны для организации поиска. Если автоматика не работает - нужен ручной режим.

### 3.4. Конкурентное обновление статусов волонтёров

**Проблема:** Два координатора одновременно пытаются назначить одного волонтёра

**Стратегия:**
1. **Блокировка:** Используем `SELECT ... FOR UPDATE` при проверке доступности волонтёров
   ```sql
   SELECT * FROM volunteers 
   WHERE id IN (1, 2, 3) AND status = 'AVAILABLE'
   FOR UPDATE;
   ```
2. **Основная транзакция №2:** ROLLBACK (группа НЕ создаётся)
3. **Retry:** Автоматически обновляем список доступных волонтёров и предлагаем координатору выбрать других
4. **Компенсация:** Заявка остаётся в статусе "NEW" (откатывается только транзакция №2)
5. **Уведомление пользователя:** "Волонтёры уже заняты. Попробуй выбрать других."

**Обоснование:** Pessimistic locking предотвращает race condition. Это критично для избежания "двойного бронирования" волонтёров.

## 4. Идемпотентность

### 4.1. Проблема: Повторное нажатие кнопки "Создать заявку"

**Сценарий:** Координатор дважды кликнул на кнопку из-за медленного интернета в полевых условиях

**Решение:**
1. **Frontend:** Блокировать кнопку после первого клика (disable button)
2. **Backend:** Проверять `idempotency_key` в запросе
   ```json
   {
     "missing_person_name": "Иван Петров",
     "coordinates": {"lat": 53.9045, "lon": 27.5615},
     "radius_km": 5,
     "idempotency_key": "coord-789_2024-11-08_09-00-15"
   }
   ```
3. **Логика:**
   - При получении запроса проверяем таблицу `idempotent_requests`
   - Если `idempotency_key` уже есть → возвращаем кэшированный ответ (request_id, group_id)
   - Если нет → обрабатываем запрос и сохраняем в `idempotent_requests`
4. **TTL ключей:** 24 часа (после этого можно повторить создание с тем же ключом)

**Пример таблицы:**
```sql
CREATE TABLE idempotent_requests (
  idempotency_key VARCHAR(128) PRIMARY KEY,
  endpoint VARCHAR(255) NOT NULL,
  response_body JSONB NOT NULL,
  created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_idempotent_created ON idempotent_requests(created_at);
-- Очистка старых записей по cron (старше 24 часов)
```

### 4.2. Проблема: Дубликаты SMS-уведомлений

**Сценарий:** Notification Service получил запрос дважды из-за retry

**Решение:**
1. В таблице `notification_queue` добавляем уникальный ключ:
   ```sql
   CREATE UNIQUE INDEX idx_notification_dedup 
   ON notification_queue(request_id, group_id, recipient_phone, notification_type);
   ```
2. При попытке добавить дубликат → игнорируем (ON CONFLICT DO NOTHING)
3. SMS-провайдер также должен поддерживать дедупликацию через `Message-ID`

### 4.3. Проблема: Повторное формирование группы для той же заявки

**Сценарий:** Координатор дважды нажал "Сформировать группу"

**Решение:**
1. Проверяем, есть ли уже группа для этой заявки и зоны:
   ```sql
   SELECT * FROM groups 
   WHERE request_id = 'REQ-2024-0142' AND assigned_zone_id = 'Z-001';
   ```
2. Если группа уже существует → возвращаем её ID (не создаём новую)
3. Если статус группы `FORMING` или `ACTIVE` → запрещаем создание новой
4. Уведомление: "Группа для этой зоны уже существует. ID: G-42"

## 6. Метрики и мониторинг

**Ключевые метрики для отслеживания:**

| Метрика | Цель | Алерт при |
|---------|------|-----------|
| `request_creation_latency_p95` | < 3 секунды | > 5 секунд |
| `request_creation_success_rate` | > 99% | < 98% |
| `group_formation_latency_p95` | < 5 секунд | > 10 секунд |
| `group_formation_success_rate` | > 98% | < 95% |
| `db_connection_pool_active` | < 80% от макс. | > 90% |
| `notification_queue_size` | < 500 | > 2000 |
| `sms_delivery_rate` | > 95% в течение 5 минут | < 90% |
| `push_delivery_rate` | > 98% в течение 1 минуты | < 95% |
| `volunteer_lock_timeout_count` | 0 | > 10/hour |
| `geo_service_timeout_count` | 0 | > 5/hour |
| `concurrent_volunteer_conflicts` | < 5/hour | > 20/hour |
| `offline_sync_pending_count` | 0 | > 50 |

**Логирование:**
- **INFO:** Успешное создание заявки и группы
- **WARNING:** Notification Service timeout, поместили в очередь
- **WARNING:** Волонтёр уже занят, откатили формирование группы
- **ERROR:** БД недоступна, откатили транзакцию
- **ERROR:** Geo Planning Service недоступен, не удалось создать зоны
- **CRITICAL:** Очередь уведомлений переполнена (> 5000)
- **CRITICAL:** Более 20% заявок не могут создать группы из-за отсутствия волонтёров

## 7. Выводы

### Ключевые решения:

1. ✅ **Две транзакции:** Создание заявки (Tx1) и формирование группы (Tx2) - разделены для гибкости
2. ✅ **Атомарность:** Заявка + зоны в одной транзакции; Группа + назначение зоны в другой
3. ✅ **Асинхронность:** SMS/Push-уведомления и запись событий - через очередь
4. ✅ **Идемпотентность:** Используем `idempotency_key` для дедупликации запросов
5. ✅ **Pessimistic Locking:** `SELECT ... FOR UPDATE` для избежания конкурентного назначения волонтёров
6. ✅ **Отказоустойчивость:** Не откатываем заявку при сбое уведомлений; Offline-режим для полевых условий
7. ✅ **Retry:** Exponential backoff для временных сбоев (SMS, Push, Geo Service)
8. ✅ **Мониторинг:** Метрики для каждого критичного компонента + алерты
9. ✅ **Eventual Consistency:** События записываются асинхронно через event sourcing

### Транзакционные инварианты:

- **Инвариант №1:** Заявка без зон недействительна → откатываем Tx1
- **Инвариант №2:** Группа не может содержать занятых волонтёров → используем pessimistic lock
- **Инвариант №3:** Зона может быть назначена только одной активной группе → unique constraint на `zones.assigned_group_id`
- **Инвариант №4:** Волонтёр не может быть в двух активных группах одновременно → проверка статуса + version для optimistic locking


### Компромиссы:

- **Eventual consistency для уведомлений:** Пользователь может получить email с задержкой, но это приемлемо
- **Синхронная проверка исполнителя:** Замедляет создание задачи, но гарантирует корректность данных
- **Хранение idempotency keys 24 часа:** Занимает место в БД, но предотвращает дубликаты

### Риски:

⚠️ **Риск 1:** Очередь уведомлений может переполниться при массовом создании задач
- **Митигация:** Установить лимит очереди + алерт при превышении 5000 сообщений

⚠️ **Риск 2:** Race condition при одновременном назначении задач одному исполнителю может превысить его лимит
- **Митигация:** Использовать SELECT FOR UPDATE при проверке лимита задач на исполнителя
