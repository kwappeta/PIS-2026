# Use-case: Создание заявки на поиск и формирование группы

**Предметная область:** Поисково-спасательный отряд «Юго-Запад» 🔦

**Первичный актор:** Координатор

**Вторичные акторы:**
- Диспетчер
- Старший группы
- Notification Service (Сервис уведомлений)
- Database (База данных)

**Цель:** Создать новую заявку на поиск пропавшего человека, разбить территорию на зоны, сформировать поисковую группу и назначить её на зону

---

## Предусловия

- Пользователь аутентифицирован и авторизован с ролью "Координатор"
- В системе есть доступные волонтёры для формирования группы
- Координаты района поиска валидны
- Сервис уведомлений доступен
- База данных доступна

---

## Основной поток (Happy Path)

1. Координатор получает звонок от диспетчера о пропавшем человеке
2. Координатор открывает форму создания заявки
3. Система отображает форму с полями: ФИО пропавшего, возраст, описание, координаты района, срочность, контакт заявителя
4. Координатор заполняет обязательные поля:
   - ФИО пропавшего: "Иван Иванов"
   - Возраст: "67 лет"
   - Описание: "Вышел из дома утром, страдает потерей памяти"
   - Координаты района: lat=53.9045, lon=27.5615
   - Радиус поиска: 5 км
   - Срочность: "URGENT"
   - Контакт заявителя: "+375291234567"
5. Координатор нажимает кнопку "Создать заявку"
6. Система проверяет валидность данных:
   - ФИО не пустое
   - Координаты валидны (широта: -90..90, долгота: -180..180)
   - Радиус поиска > 0 и < 100 км
   - Телефон соответствует формату
7. Система создаёт запись заявки в БД со статусом "NEW"
8. Система генерирует уникальный ID заявки (например, "REQ-2024-0142")
9. Система автоматически разбивает территорию на 4 зоны (North, South, East, West)
10. Система публикует доменное событие "RequestCreated"
11. Координатор открывает форму формирования группы
12. Система отображает доступных волонтёров (статус "AVAILABLE")
13. Координатор выбирает старшего группы "Петров П.П." и 3 волонтёров
14. Координатор указывает чек-лист снаряжения: GPS, рация, аптечка, фонарь
15. Система проверяет доступность волонтёров
16. Система создаёт группу "G-42" со статусом "FORMING"
17. Система назначает группу на зону "North"
18. Система обновляет статус заявки на "IN_PROGRESS"
19. Система обновляет статус зоны "North" на "ASSIGNED"
20. Система публикует событие "GroupFormed" и "GroupAssignedToZone"
21. Система отправляет push-уведомления всем участникам группы
22. Notification Service отправляет SMS старшему группы с координатами
23. Система возвращает успешный ответ
24. Координатор видит сообщение "Заявка #REQ-2024-0142 создана. Группа G-42 назначена на зону North. Участники уведомлены."
25. Система отображает карту с зонами и назначенной группой

---

## Постусловия

- Заявка создана в базе данных со статусом "IN_PROGRESS"
- Территория разбита на 4 зоны (North, South, East, West)
- Группа сформирована со статусом "FORMING"
- Зона "North" имеет статус "ASSIGNED" и ссылку на группу
- Все участники группы получили уведомления
- Старший группы получил SMS с координатами
- События "RequestCreated", "GroupFormed", "GroupAssignedToZone" записаны в event log
- Заявка отображается в журнале активных заявок

## Альтернативные потоки

### 4a. Координатор не заполнил обязательное поле

**Условие:** На шаге 4 координатор оставил пустым поле "ФИО пропавшего"

**Поток:**
1. 4a1. Координатор нажимает "Создать заявку"
2. 4a2. Система подсвечивает пустое поле красным
3. 4a3. Система показывает сообщение "ФИО пропавшего обязательно для заполнения"
4. 4a4. Возврат к шагу 4

### 5a. Координатор хочет отложить создание

**Условие:** На любом шаге до нажатия "Создать заявку"

**Поток:**
1. 5a1. Координатор нажимает кнопку "Сохранить черновик"
2. 5a2. Система сохраняет введённые данные со статусом "DRAFT"
3. 5a3. Система показывает сообщение "Черновик сохранён. ID: DRAFT-42"
4. 5a4. Координатор может вернуться к редактированию позже
5. 5a5. Use-case завершается успешно

### 13a. Нет доступных волонтёров

**Условие:** На шаге 13 система не находит волонтёров со статусом "AVAILABLE"

**Поток:**
1. 13a1. Система показывает сообщение "Нет доступных волонтёров для формирования группы"
2. 13a2. Координатор может:
   - Отложить формирование группы (заявка остаётся в статусе "NEW")
   - Изменить срочность на "CRITICAL" и запросить привлечение резервных волонтёров
3. 13a3. Если координатор выбрал "Отложить":
   - Система сохраняет заявку со статусом "PENDING_GROUP"
   - Система публикует событие "GroupFormationPending"
4. 13a4. Use-case завершается с частичным успехом (заявка создана, группа не сформирована)

---

## Исключительные ситуации

### 6a. Координаты района не валидны

**Условие:** На шаге 6 система обнаруживает, что координаты выходят за допустимые пределы

**Поток:**
1. 6a1. Система возвращает ошибку валидации "Координаты должны быть в диапазоне: lat[-90,90], lon[-180,180]"
2. 6a2. Система НЕ создаёт заявку
3. 6a3. Система показывает сообщение "Координаты указаны некорректно. Проверь GPS! 🧭"
4. 6a4. Use-case завершается неудачей

### 6b. Радиус поиска слишком большой

**Условие:** На шаге 6 система обнаруживает, что радиус > 100 км

**Поток:**
1. 6b1. Система возвращает ошибку "Радиус поиска не может превышать 100 км"
2. 6b2. Система НЕ создаёт заявку
3. 6b3. Система показывает сообщение "Радиус слишком большой. Для больших территорий создай несколько заявок."
4. 6b4. Координатор корректирует радиус
5. 6b5. Возврат к шагу 5

### 7a. База данных недоступна

**Условие:** На шаге 7 БД не отвечает

**Поток:**
1. 7a1. Система получает Database Connection Timeout Exception
2. 7a2. Система логирует ошибку с деталями и timestamp
3. 7a3. Система откатывает транзакцию (rollback)
4. 7a4. Система показывает сообщение "Не удалось создать заявку. Попробуй ещё раз или свяжись с техподдержкой."
5. 7a5. Use-case завершается неудачей

### 9a. Не удалось разбить территорию на зоны

**Условие:** На шаге 9 алгоритм разбиения вернул ошибку (например, пересечение с запретной зоной)

**Поток:**
1. 9a1. Система получает исключение от модуля геопланирования
2. 9a2. Система откатывает создание заявки
3. 9a3. Система показывает сообщение "Не удалось разбить район на зоны. Возможно, территория пересекается с запретной зоной."
4. 9a4. Система предлагает координатору скорректировать координаты или радиус
5. 9a5. Use-case завершается неудачей

### 15a. Выбранные волонтёры уже заняты

**Условие:** На шаге 15 система обнаруживает, что волонтёры изменили статус на "BUSY" (конкурентное обновление)

**Поток:**
1. 15a1. Система возвращает ошибку "Один или несколько волонтёров уже назначены на другую заявку"
2. 15a2. Система откатывает формирование группы (транзакция на создание группы)
3. 15a3. Система НЕ откатывает создание заявки (заявка остаётся в статусе "NEW")
4. 15a4. Система показывает сообщение "Волонтёры уже заняты. Попробуй выбрать других."
5. 15a5. Координатор возвращается к шагу 13 и выбирает других волонтёров
6. 15a6. Use-case продолжается с шага 13

### 21a. Notification Service недоступен

**Условие:** На шаге 21 Notification Service не отвечает в течение 10 секунд

**Поток:**
1. 21a1. Система получает Timeout Exception от Notification Service
2. 21a2. Система НЕ откатывает создание заявки и группы (данные уже в БД)
3. 21a3. Система помещает уведомления в очередь для отложенной отправки
4. 21a4. Система логирует "NotificationService timeout, notifications queued for retry"
5. 21a5. Система публикует событие "NotificationFailed" с деталями для мониторинга
6. 21a6. Система возвращает успешный ответ (заявка и группа созданы)
7. 21a7. Координатор видит сообщение "Заявка создана! Уведомления будут отправлены позже. Можно позвонить старшему вручную."
8. 21a8. Use-case завершается успешно (с предупреждением)

### 22a. SMS-сервис вернул ошибку

**Условие:** На шаге 22 SMS не удалось отправить (invalid number, quota exceeded, etc.)

**Поток:**
1. 22a1. Notification Service возвращает ошибку отправки SMS
2. 22a2. Система НЕ откатывает создание заявки и группы
3. 22a3. Система помечает SMS как "failed" и ставит в очередь для повторной попытки
4. 22a4. Система логирует детали ошибки (номер телефона, код ошибки, timestamp)
5. 22a5. Система отправляет push-уведомление как fallback (если доступно)
6. 22a6. Система возвращает успешный ответ
7. 22a7. Координатор видит предупреждение "SMS не отправлена. Свяжись со старшим группы по телефону."
8. 22a8. Use-case завершается успешно (с предупреждением)

## Частота использования

- **Средняя** - несколько раз в день (в зависимости от сезона и региона)
- **Пиковая нагрузка** - до 10-15 заявок в день в сезон «грибников» или при ЧП

## Критические требования

- **Производительность:** Создание заявки должно занимать < 3 секунд
- **Производительность:** Формирование группы должно занимать < 5 секунд
- **Доступность:** 99.5% uptime (система должна работать даже в условиях плохого интернета)
- **Идемпотентность:** Повторное нажатие "Создать" не должно создавать дубликаты
- **Безопасность:** Только пользователи с ролью "Координатор" могут создавать заявки
- **Безопасность:** Данные заявок (ФИО, контакты) должны быть зашифрованы
- **Аудит:** Все действия логируются с timestamp и user_id для отслеживания
- **Геоданные:** Координаты должны валидироваться и очищаться от некорректных значений
- **Offline-режим:** Заявка должна сохраняться локально и синхронизироваться при восстановлении связи

## Открытые вопросы

1. Что делать, если координаты района пересекаются с запретной зоной (военный объект, АЭС)?
2. Нужно ли автоматически формировать группу на основе доступности и компетенций волонтёров?
3. Должна ли система поддерживать создание заявок по телефону (голосовой ввод)?
4. Как обрабатывать ситуацию, когда заявка отменяется после формирования группы?
5. Нужно ли отправлять уведомления родственникам пропавшего?
6. Должен ли быть лимит на количество активных заявок, которые может вести один координатор?
7. Как обрабатывать случай, когда заявка дублируется (тот же человек, та же территория)?
