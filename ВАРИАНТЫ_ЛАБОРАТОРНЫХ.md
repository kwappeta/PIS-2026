# Варианты лабораторных работ по дисциплине "Проектирование интернет-систем"

## Общая концепция

Каждый студент выбирает **один вариант** предметной области и последовательно развивает его через все 9 лабораторных работ. Каждая последующая работа усложняет и расширяет предыдущую, формируя единую архитектуру системы.

## Прогрессия развития системы (на примере Варианта 1: Таск-трекер «Не прокрастинируй» 😎)

### Лабораторная работа 1. Сценарий транзакции
**Цель:** Моделирование основного use-case и границ ответственности

**Задачи:**
- Описать сценарий "Создание задачи с назначением": выбор проекта → описание задачи → назначение исполнителя → установка дедлайна → уведомление
- Построить диаграмму последовательности (sequence diagram)
- Определить акторов (Создатель задачи, Исполнитель, Система уведомлений, Email-сервис)
- Выявить граничные условия: исполнитель не найден, дедлайн в прошлом, превышен лимит задач
- Зафиксировать точки отказа и механизмы восстановления

**Артефакты:**
- Use-case диаграмма с молодежным подходом к описанию 🎯
- Sequence diagram с приколами в комментариях
- Описание сценариев успеха и фейлов в формате Gherkin (Given-When-Then)
- README с обоснованием границ ответственности (и почему "на потом" - плохо)

---

### Лабораторная работа 2. Гексагональная архитектура
**Цель:** Спроектировать архитектуру основного сервиса с изоляцией домена (чтобы не всё в кучу 😅)

**Задачи:**
- Выбрать центральный сервис (например, "Управление задачами")
- Спроектировать структуру проекта с разделением на слои:
  - `domain/` - чистая бизнес-логика (никаких фреймворков, только хардкор!)
  - `application/` - сценарии использования (что можно делать с задачами)
  - `infrastructure/` - адаптеры (БД, HTTP, очереди и прочий инфра-треш)
- Определить **порты** (интерфейсы - контракты без наеба):
  - Входящие: `ITaskService`, `IProjectService`
  - Исходящие: `ITaskRepository`, `INotificationService`
- Реализовать **адаптеры** (реальные имплементации):
  - `InMemoryTaskRepository` (пока без БД, просто Map)
  - `RestTaskController` (крутой REST API)

**Артефакты:**
- Код на выбранном языке (Java/C#/Python/Node.js/Go - что душе угодно)
- Диаграмма архитектуры (шестиугольник power! 💪)
- README с описанием портов и адаптеров (и почему это не overengineering)

### Лабораторная работа 3. Доменный уровень
**Цель:** Разработать полноценную доменную модель с агрегатами и бизнес-правилами (настоящий DDD, а не просто сеттеры/геттеры 🔥)

**Задачи:**
- Определить **сущности** (Entities - те, что с ID):
  - `Task` (ID, название, описание, статус, приоритет)
  - `Project` (ID, название, лимит задач)
  - `User` (ID, имя, email, роль)
- Определить **Value Objects** (иммутабельные штуки):
  - `TaskStatus` (todo/in-progress/done)
  - `Priority` (low/medium/high/urgent)
  - `Deadline` (дата + валидация что не в прошлом)
  - `AssignmentInfo` (исполнитель, дата назначения)
- Спроектировать **Агрегат Task**:
  - Корень агрегата: `Task`
  - Инварианты: нельзя закрыть таску без исполнителя; дедлайн не в прошлом; нельзя менять задачу после архивации
  - Методы: `assign()`, `updateStatus()`, `postponeDeadline()`, `archive()`
- Реализовать доменные события: `TaskCreated`, `TaskAssigned`, `TaskCompleted`, `DeadlineMissed`

**Артефакты:**
- Код доменных классов с настоящей бизнес-логикой (не CRUD! 💪)
- Юнит-тесты инвариантов (например, нельзя назначить таск тому, кто уволился)
- Диаграмма классов доменной модели
- README с описанием агрегатов и правил (и почему Entity ≠ таблица в БД)

### Лабораторная работа 4. Уровень приложения
**Цель:** Реализовать сценарии использования с CQRS на уровне интерфейса (команды пишут, запросы читают - никакой путаницы! 📝📖)

**Задачи:**
- Разделить операции на **команды** (изменяют состояние) и **запросы** (просто смотрят):
  - Команды: `CreateTaskCommand`, `AssignTaskCommand`, `CompleteTaskCommand`
  - Запросы: `GetTaskByIdQuery`, `GetMyTasksQuery`, `GetProjectStatsQuery`
- Реализовать **обработчики команд** (Command Handlers - делают дело):
  - `CreateTaskHandler`: валидация → создание агрегата → сохранение → ивент "TaskCreated"
  - `AssignTaskHandler`: проверка что юзер существует → назначение → нотификация
- Реализовать **обработчики запросов** (Query Handlers - просто показывают данные):
  - `GetTaskByIdHandler`: достать таску из репозитория
  - `GetMyTasksHandler`: фильтр по текущему юзеру + статусу
- Добавить валидацию на уровне приложения (емейл валидный? дедлайн не в прошлом?)

**Артефакты:**
- Код команд, запросов и их обработчиков (каждый знает свое дело)
- REST API эндпоинты:
  - `POST /api/tasks` (CreateTask)
  - `PUT /api/tasks/{id}/assign` (AssignTask)
  - `GET /api/tasks/my` (GetMyTasks)
- Интеграционные тесты обработчиков
- README с описанием сценариев (и почему CQRS - это круто)

### Лабораторная работа 5. Инфраструктурный уровень
**Цель:** Реализовать персистентность, интеграции и конфигурацию (время подключать реальную БД! 🗄️)

**Задачи:**
- Реализовать **репозитории** с настоящей БД (PostgreSQL/MySQL - не SQLite, ну пожалуйста 🙏):
  - `TaskRepository`: сохранение/загрузка агрегатов Task
  - `ProjectRepository`: управление проектами
  - Миграции схемы БД (через Flyway/Liquibase/Alembic - без прямых ALTER TABLE!)
- Реализовать **адаптеры внешних сервисов**:
  - `EmailNotificationAdapter` (отправка уведомлений - для начала просто логи или Mailtrap)
  - `SlackNotificationAdapter` (опционально - шлём алерты в Slack, если умеете)
- Добавить **конфигурацию**:
  - Dependency Injection контейнер (Spring/ASP.NET Core/NestJS - без new внутри кода!)
  - Конфигурация подключений (БД, очереди, внешние API)
  - Переменные окружения для секретов (пароли в `.env`, не в git! 🔐)

**Артефакты:**
- Код репозиториев и адаптеров (всё через интерфейсы)
- SQL-миграции (создание таблиц tasks, projects, users)
- Docker Compose для локального окружения (БД + может Redis для кэша)
- README с инструкциями по запуску (чтобы одной командой всё поднималось)

  - Обработчики команд: мокирование зависимостей (репозитории, адаптеры)
  - Покрытие не менее 80% для доменного слоя (но качество важнее количества!)
- **Интеграционные тесты** (Integration Tests - проверяем что всё вместе работает):
  - Тестирование репозиториев с тестовой БД (Testcontainers - огонь! 🔥)
  - Тестирование REST API эндпоинтов (полный цикл запрос-ответ)
  - Проверка транзакционности операций (откатывается ли при ошибке?)
- **Контрактные тесты** (опционально, но круто):
  - Проверка соответствия API контракту (Pact/Postman Collections)
- Настроить CI pipeline (GitHub Actions/GitLab CI - автоматизация рулит!):
  - Прогон тестов при каждом коммите (красный CI = не мерджим!)
  - Проверка покрытия кода

**Артефакты:**
- Комплект тестов (unit + integration)
- Отчёт покрытия кода (JaCoCo/Coverage.py/Istanbul)
- CI конфигурация (.github/workflows/test.yml)
- README с описанием стратегии тестирования (и почему тесты - это инвестиция)

### Лабораторная работа 7. CQRS
**Цель:** Разделить модели чтения и записи для повышения производительности (пишем в одно, читаем из другого! 🚀)

**Задачи:**
- Реализовать **Command Model** (модель записи - нормализованная БД):
  - Агрегаты Task, Project в нормализованной схеме БД
  - Обработка команд через `TaskCommandHandler`
- Реализовать **Query Model** (модель чтения - денормализованная для скорости):
  - Денормализованные проекции для быстрых запросов:
    - `TaskListView` (id, название, статус, исполнитель, дедлайн - всё в одной таблице)
    - `ProjectDashboard` (id проекта, кол-во задач, кол-во завершённых, прогресс %)
  - Отдельная read-only БД или материализованные представления (views)
- Обеспечить **Eventual Consistency** (данные придут... через секунду... ну или две 😅):
  - При создании задачи публикуется событие `TaskCreated`
  - Обработчик события обновляет проекцию `TaskListView`
  - Проверить задержку синхронизации (с метриками)
- Добавить кэширование запросов (Redis) для популярных проекций (скорость light! ⚡)

**Артефакты:**
- Код Command Handlers и Query Handlers (разные модели!)
- Схема БД для write model и read model
- Обработчики доменных событий для обновления проекций
- README с объяснением eventual consistency (и почему это норм)

### Лабораторная работа 8. Модули и сервисы
**Цель:** Декомпозировать монолит на bounded contexts и сервисы (разделяй и... не властвуй, а балансируй! 🎭)

**Задачи:**
- Выделить **Bounded Contexts** (каждый знает своё дело):
  1. **Task Service** (управление задачами)
  2. **Project Service** (управление проектами)
  3. **User Service** (управление пользователями, права)
  4. **Notification Service** (отправка уведомлений)
- Определить **контракты взаимодействия**:
  - REST API между сервисами (синхронное - когда нужен ответ прямо сейчас)
  - Доменные события через message broker (асинхронное - отправил и забыл):
    - `TaskAssigned` → Notification Service шлёт уведомление
    - `DeadlineApproaching` → Notification Service напоминает исполнителю
- Реализовать **API Gateway** (опционально, но по-взрослому):
  - Единая точка входа для клиентов (один URL для всех сервисов)
  - Маршрутизация запросов к сервисам
- Обеспечить **отказоустойчивость** (чтобы всё не рухнуло от одной ошибки):
  - Retry-логика при сбоях (попробуй ещё раз!)
  - Circuit Breaker для внешних вызовов (если не отвечает - не долби)

**Артефакты:**
- Код 3-4 микросервисов (каждый в отдельной папке/репозитории)
- OpenAPI (Swagger) спецификации для каждого сервиса
- Docker Compose для локального запуска всех сервисов (одна команда - все сервисы up!)
- Диаграмма взаимодействия сервисов (context map)
- README с описанием границ контекстов (и почему не делаем один большой сервис)

### Лабораторная работа 9. Буферы протоколов (Protocol Buffers)
**Цель:** Использовать Protocol Buffers и gRPC для межсервисного взаимодействия (бинарный протокол - быстро и типобезопасно! ⚡)

**Задачи:**
- Определить **схемы .proto** для основных сущностей:
  ```protobuf
  message Task {
    string id = 1;
    string title = 2;
    string description = 3;
    TaskStatus status = 4;
    string assignee_id = 5;
    int64 deadline_unix = 6;
  }
  
  message CreateTaskRequest {
    string project_id = 1;
    string title = 2;
    string description = 3;
  }
  
  message CreateTaskResponse {
    string task_id = 1;
    TaskStatus status = 2;
  }
  ```
- Реализовать **gRPC сервисы** (быстрее REST, меньше траффика):
  - `TaskService.CreateTask(CreateTaskRequest) -> CreateTaskResponse`
  - `TaskService.AssignTask(AssignTaskRequest) -> Task`
- Обеспечить **обратную совместимость** (чтобы старые клиенты не сломались):
  - Добавить новое поле в схему без breaking changes (например, `priority`)
  - Версионирование схем (v1, v2 - или через package name)
  - Протестировать старый клиент с новым сервером (должно работать!)
- Сгенерировать клиенты и серверы для выбранного языка (код пишется автоматически! 🤖)
- Сравнить производительность gRPC vs REST (бенчмарк - кто быстрее?)

**Артефакты:**
- .proto файлы для всех сервисов
- Код gRPC серверов и клиентов (сгенерированный + бизнес-логика)
- Документация по версионированию схем (что можно менять, что нельзя)
- Результаты бенчмарка (latency, throughput - с графиками!)
- README с инструкциями по генерации кода (одна команда - весь код готов)
